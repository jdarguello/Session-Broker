# Bootstrapping the Cluster

This page describes how to bring up a fresh cluster with all Session Broker dependencies deployed via Argo CD.

## Container image (CI)

The session-broker image is built and published to the **GitHub Container Registry (GHCR)** by the
[`deploy-image.yml`](https://github.com/jdarguello/Session-Broker/blob/main/.github/workflows/deploy-image.yml)
workflow. It runs on pushes to `main` that touch `src/**` (and on manual `workflow_dispatch`).

- **Image:** `ghcr.io/jdarguello/session-broker`
- **Tags:** `latest` and `sha-<commit>` (immutable, for pinning)
- **Platforms:** `linux/amd64` **and** `linux/arm64` (multi-arch) вҖ” so the image
  pulls on Apple-Silicon/arm64 nodes as well as amd64. An amd64-only image is what
  causes `ErrImagePull` (`no matching manifest for linux/arm64`) on arm64 clusters.
- **Auth:** the built-in `GITHUB_TOKEN` (`packages: write`) вҖ” no extra secrets to configure
- **Build context:** `src/` (uses `src/Dockerfile`), with GitHub Actions layer caching

The GitOps manifests reference this image: the `base` deployment pins `:latest`, while the `dev` and
`prod` overlays override the tag via Kustomize `images:` (dev вҶ’ `latest`, prod вҶ’ a pinned release tag).

> The GHCR package must be **public**, or the cluster needs an `imagePullSecret` for `ghcr.io`,
> otherwise the kubelet cannot pull the image.

## Prerequisites

- `kubectl` connected to the target cluster
- Argo CD installed in the `argocd` namespace ([installation docs](https://argo-cd.readthedocs.io/en/stable/getting_started/))
- A **default `StorageClass`** with a working dynamic provisioner (see below)
- Secrets created before syncing (see below)

## Cluster storage (default StorageClass)

Several dependencies request `PersistentVolumeClaim`s вҖ” Dapr's **scheduler** (etcd
data dir), **Redis** (`master`), and **Keycloak**'s bundled **PostgreSQL**. A bare
`kubeadm` cluster ships with **no** storage provisioner and **no** default
`StorageClass`, so every one of those PVCs stays `Pending` and the pods never
schedule:

```
FailedScheduling  pod has unbound immediate PersistentVolumeClaims
FailedBinding     no persistent volumes available for this claim and no storage class is set
```

For a lab/demo cluster, install [local-path-provisioner](https://github.com/rancher/local-path-provisioner)
(dynamic `hostPath` volumes) and mark it default:

```bash
kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/v0.0.36/deploy/local-path-storage.yaml
kubectl annotate storageclass local-path storageclass.kubernetes.io/is-default-class=true --overwrite
```

Since Kubernetes 1.28 the `RetroactiveDefaultStorageClass` feature assigns the new
default to any already-`Pending` PVCs automatically вҖ” no need to delete/recreate them.

## Dependency images (Bitnami / `bitnamilegacy`)

The Redis and Keycloak Helm values pin their images to **`docker.io/bitnamilegacy/*`**
rather than `docker.io/bitnami/*`. On **2025-08-28** Broadcom's "Bitnami Secure Images"
migration removed the free `docker.io/bitnami/*` **versioned** tags (only a small set of
`latest` tags remain). Pulls of the chart-default tags now fail with:

```
failed to resolve reference "docker.io/bitnami/redis:<tag>": not found  (ErrImagePull / NotFound)
```

The retired tags were copied verbatim to the frozen **`docker.io/bitnamilegacy/*`** repo,
so the values just repoint the registry/repository there. **`bitnamilegacy` is
unmaintained** (no further updates or CVE fixes) вҖ” acceptable for the demo, but a
production deployment should migrate to maintained images.

## Required secrets

Create these Kubernetes secrets before Argo CD syncs the apps:

```bash
# Redis password (referenced by both the Redis chart and the Dapr state store component)
kubectl create secret generic redis-secret \
  --from-literal=redis-password=<your-password> \
  -n redis --dry-run=client -o yaml | kubectl apply -f -

# Copy the secret into session-broker namespace for the Dapr component
kubectl create secret generic redis-secret \
  --from-literal=redis-password=<your-password> \
  -n session-broker --dry-run=client -o yaml | kubectl apply -f -

# Keycloak admin password
kubectl create secret generic keycloak-admin-secret \
  --from-literal=admin-password=<your-password> \
  -n keycloak --dry-run=client -o yaml | kubectl apply -f -

# OAuth confidential-client secret (write path). The broker sources
# KEYCLOAK_CLIENT_SECRET from this Secret via secretKeyRef. The value MUST equal
# the secret generated for the `session-broker` client in the Keycloak realm
# (see "Keycloak realm setup" below). Missing this Secret is a hard failure:
# the pod stays in CreateContainerConfigError / CrashLoopBackOff.
kubectl create secret generic session-broker-secret \
  --from-literal=keycloak-client-secret=<client-secret-from-keycloak> \
  -n session-broker --dry-run=client -o yaml | kubectl apply -f -
```

### Token encryption key (demo gap)

The broker encrypts the cached token set with AES-256-GCM and requires
`TOKEN_ENCRYPTION_KEY` (base64 of a 32-byte key). **For the ArgoCon demo this is a
plaintext throwaway value committed directly in the Deployment env** ([`gitops/session-broker/base/deployment.yaml`](https://github.com/jdarguello/Session-Broker/blob/main/gitops/session-broker/base/deployment.yaml)),
so no extra secret is needed to bring the writer/reader up.

This is an **explicitly accepted demo gap** вҖ” the same category as the trusted
`POST /auth/callback/cache` write path. At go-live the key moves out of the
manifest into a pre-created `session-broker-secret` consumed via `secretKeyRef`,
and the committed value is rotated out.

## Bootstrap

Pick **one** of the two equivalent entry points (do not apply both вҖ” they create the same four Applications and would collide).

**Option A вҖ” app-of-apps root:**

```bash
kubectl apply -f gitops/apps/root-app.yaml
```

**Option B вҖ” ApplicationSet (single file):**

```bash
kubectl apply -f gitops/bootstrap.yaml
```

Either way, Argo CD discovers and syncs all child Applications (`dapr`, `keycloak`, `redis`, `session-broker`). The ApplicationSet requires the **applicationset-controller**, which ships with the default Argo CD install.

## Verify

```bash
argocd app list
kubectl get pods -n session-broker
kubectl get pods -n dapr-system
kubectl get pods -n keycloak
kubectl get pods -n redis
```

## Keycloak realm setup

The broker is a **confidential OAuth client** that performs the `codeвҶ’token`
exchange itself (see [Identity Correlation & Write Path](../architecture/dapr-keycloak)).
After Keycloak is running, configure the realm via the Admin UI or `kcadm.sh`:

1. **Realm:** `enterpriseclaw`.
2. **Client:** `session-broker` вҖ” **confidential** (client authentication ON),
   standard flow ON, PKCE `S256`. Register both redirect URIs:
   - `https://broker.session-broker.example.com/auth/callback` (prod / ALB)
   - `http://localhost:8000/auth/callback` (local tunnel вҖ” see Stage B below)

   Copy the generated client secret into the `session-broker-secret` Secret
   (`keycloak-client-secret` key) created above.
3. **Identity provider (optional, for the Google demo narrative):** add a
   `google` OpenID Connect IdP. Its broker redirect URI вҖ”
   `http://localhost:8080/realms/enterpriseclaw/broker/google/endpoint` (local)
   вҖ” must be registered byte-for-byte in the Google Cloud OAuth client.
4. **Test user:** a local realm user (e.g. `alice`) with a password is enough to
   exercise the flow without Google.

> The Dapr **bearer middleware is intentionally excluded** from the demo pipeline
> (see [`base/kustomization.yaml`](https://github.com/jdarguello/Session-Broker/blob/main/gitops/session-broker/base/kustomization.yaml));
> the broker validates provenance by minting tokens via the `code` exchange, not
> by validating an inbound bearer.

## Stage B: running the OAuth write path locally (no AWS ALB)

The production write path exposes `GET /auth/callback` through an **AWS ALB
Ingress**, which has no address on the local kubeadm-in-UTM cluster. You do **not**
need a cloud load balancer to run the full browser flow locally вҖ” only the user's
**browser** ever dereferences the front-channel URLs (OAuth redirects are HTTP
`302`s to the user-agent, never server-to-server calls). An **SSH tunnel** from the
Mac to the cluster is sufficient.

**1. Front-channel env (committed in the dev overlay).**
[`overlays/dev/patch-deployment.yaml`](https://github.com/jdarguello/Session-Broker/blob/main/gitops/session-broker/overlays/dev/patch-deployment.yaml)
points the broker at `localhost` for the two browser-facing URLs, while the
back-channel `codeвҶ’token` exchange keeps the in-cluster `KEYCLOAK_TOKEN_URL`
inherited from base:

| Var | Value (dev) | Channel |
|---|---|---|
| `KEYCLOAK_ISSUER_URL` | `http://localhost:8080/realms/enterpriseclaw` | front (browser) |
| `KEYCLOAK_REDIRECT_URI` | `http://localhost:8000/auth/callback` | front (browser) |
| `KEYCLOAK_TOKEN_URL` | `http://keycloak.keycloak.svc.cluster.local/realms/enterpriseclaw/protocol/openid-connect/token` | back (broker pod) |

The broker does **not** validate the token `iss`, so the front-channel
(`localhost`) vs back-channel (`svc.cluster.local`) host mismatch is harmless.

**2. Tunnel Keycloak and the broker to the Mac browser.** Map the Mac's
`localhost:8080` to Keycloak and `localhost:8000` to the broker (ports must match
the env/redirect-URI registrations exactly):

```bash
# On the cluster node (binds the two services to the node's localhost):
ssh controlplane kubectl port-forward -n keycloak       svc/keycloak       8080:80 &
ssh controlplane kubectl port-forward -n session-broker svc/session-broker 8000:80 &
# Bridge them to the Mac:
ssh -N -L 8080:localhost:8080 -L 8000:localhost:8000 controlplane
```

**3. Drive the flow:**

```bash
# 1) Mint the binding + authorize URL (internal-only endpoint)
curl -s -X POST http://localhost:8000/auth/login/start \
  -H 'content-type: application/json' -d '{"slack_user_id":"U-DEMO-001"}'
# 2) Open the returned authorize URL in the Mac browser, log in
#    (local user `alice`, or "Sign in with Google" once the IdP is wired).
# 3) The browser lands on http://localhost:8000/auth/callback and the broker
#    stores the encrypted token set keyed by slack:U-DEMO-001.
# 4) Resolve the cached access token + claims (Slack ID is passed as a header):
curl -s -X POST http://localhost:8000/identity/resolve \
  -H 'X-Slack-User-Id: U-DEMO-001'
```

The `localhost` URLs work because the browser and the tunnel endpoints live on the
same machine (the Mac). The only hops that need real network reachability are the
two back-channel calls вҖ” KeycloakвҶ’Google (needs node egress) and brokerвҶ’Keycloak
(in-cluster) вҖ” neither of which targets `localhost`.
