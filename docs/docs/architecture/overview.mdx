# Architecture Overview

Session Broker is a single-responsibility microservice deployed on Kubernetes with a Dapr sidecar and integrated with Keycloak, Redis, and Istio.

The broker has two explicit responsibilities:

1. Write path: cache tokens after Keycloak callback.
2. Read path: resolve cached token context and convert it to workflow identity.

## Runtime flows

### A) Write flow: Keycloak callback -> cache

1. User begins authentication from Slack/Teams interaction through external gateway.
2. User authenticates in Google IdP via Keycloak.
3. Keycloak calls Session Broker callback endpoint directly.
4. Broker encrypts raw token set (access, refresh, ID token) and writes to Redis.
5. Redis key is Slack user ID; TTL is taken from token `exp` claim.
6. Broker emits audit log for write.

### B) Read flow: event -> identity conversion

1. Chat event arrives with Slack user ID.
2. Broker reads cached token set from Redis by Slack user ID.
3. Broker verifies token is not expired (`exp` strict policy).
4. Broker resolves effective authorization context.
5. Broker returns workflow identity contract and mints/requests SPIFFE/Istio token if integrated.
6. Broker emits audit log for read.

## Responsibilities

| Layer | Responsibility |
|---|---|
| Keycloak | Authenticates users and sends callback payload to broker |
| Dapr sidecar + state API | Service invocation and Redis abstraction |
| FastAPI broker | Callback processing, token cache write/read, identity conversion contract |
| Redis | Token cache keyed by Slack user ID with exp-based TTL |
| Istio/SPIFFE | Workflow identity target consumed by downstream services |
| Argo CD | Delivers manifests from `gitops/` to the cluster |
