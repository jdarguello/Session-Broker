# Dapr + Keycloak Integration

Session Broker uses Keycloak as token issuer and Dapr as runtime middleware plus state abstraction.

## Current integration (implemented)

### 1. Dapr bearer middleware validates incoming tokens

The Session Broker does not validate JWT signatures itself. Dapr's `middleware.http.bearer` component validates signature, issuer, audience, and expiry before requests reach FastAPI.

Middleware component:

```yaml
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: keycloak-validator
  namespace: session-broker
spec:
  type: middleware.http.bearer
  version: v1
  metadata:
    - name: jwksURL
      value: "http://keycloak.keycloak.svc.cluster.local/realms/session-broker/protocol/openid-connect/certs"
    - name: audience
      value: "session-broker-client"
    - name: issuer
      value: "http://keycloak.keycloak.svc.cluster.local/realms/session-broker"
```

### 2. Broker decodes claims for app-level identity fields

After Dapr validation succeeds, the broker extracts these claims from JWT payload:

- `sub` -> `user_id`
- `email` -> `user_email`
- `realm_access.roles` -> `roles`

This identity data is attached to session documents.

### 3. Dapr state component handles Redis I/O

`gitops/session-broker/base/dapr-statestore.yaml` defines the state store used by the broker:

```yaml
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: redis
  namespace: session-broker
spec:
  type: state.redis
  version: v1
  metadata:
    - name: redisHost
      value: "redis-master.redis.svc.cluster.local:6379"
```

### 4. Current API surface behind this integration

The currently implemented authenticated API surface is:

```http
POST   /sessions
GET    /sessions/{session_id}
PATCH  /sessions/{session_id}
POST   /sessions/{session_id}/handoff
DELETE /sessions/{session_id}
GET    /health
```

## Dapr invocation support

The current endpoints can also be called through Dapr service invocation:

```http
POST   /v1.0/invoke/session-broker/method/sessions
GET    /v1.0/invoke/session-broker/method/sessions/{session_id}
PATCH  /v1.0/invoke/session-broker/method/sessions/{session_id}
POST   /v1.0/invoke/session-broker/method/sessions/{session_id}/handoff
DELETE /v1.0/invoke/session-broker/method/sessions/{session_id}
GET    /v1.0/invoke/session-broker/method/health
```

## Keycloak realm setup

After Keycloak is deployed, configure the following (can be scripted via the Admin REST API):

| Setting | Value |
|---|---|
| Realm | `session-broker` |
| Client ID | `session-broker-client` |
| Client scopes | Include scopes required by callers of `/sessions` endpoints |
| Realm roles | Roles mapped into `realm_access.roles` claims |
| Validation target | Dapr middleware validates bearer token before app logic |

Keycloak's JWKS endpoint will be available at:
```
http://keycloak.keycloak.svc.cluster.local/realms/session-broker/protocol/openid-connect/certs
```
