# GitOps Overview

All Kubernetes infrastructure is managed declaratively through the `gitops/` directory and delivered by **Argo CD** using the **app-of-apps** pattern.

## Directory structure

```
gitops/
  bootstrap.yaml               ← ApplicationSet (single-file alternative to app-of-apps)

  apps/                        ← Argo CD Application manifests
    root-app.yaml              ← Root app (syncs this folder)
    session-broker.yaml        ← Kustomize overlay for the service
    dapr.yaml                  ← Dapr Helm chart
    keycloak.yaml              ← Bitnami Keycloak Helm chart
    redis.yaml                 ← Bitnami Redis Helm chart

  session-broker/
    base/                      ← Shared K8s manifests
      deployment.yaml
      service.yaml
      serviceaccount.yaml
      dapr-statestore.yaml     ← Dapr Component: state.redis
      dapr-bearer-middleware.yaml  ← Dapr Component: middleware.http.bearer
      dapr-configuration.yaml  ← Dapr Configuration (HTTP pipeline)
      kustomization.yaml
    overlays/
      dev/                     ← 1 replica, low resources, latest image tag
      prod/                    ← 3 replicas, higher resources, pinned image tag

  dapr/values.yaml             ← Helm values for Dapr
  keycloak/values.yaml         ← Helm values for Bitnami Keycloak
  redis/values.yaml            ← Helm values for Bitnami Redis
```

## App-of-apps pattern

```
Argo CD
  └── root-app  (watches gitops/apps/)
        ├── session-broker  ──▶ gitops/session-broker/overlays/dev
        ├── dapr            ──▶ dapr/dapr Helm chart
        ├── keycloak        ──▶ bitnami/keycloak Helm chart
        └── redis           ──▶ bitnami/redis Helm chart
```

The root Application is bootstrapped once manually (or via `kubectl apply`). After that, every change pushed to `main` is picked up by Argo CD automatically.

## ApplicationSet alternative (`bootstrap.yaml`)

`gitops/bootstrap.yaml` is an **ApplicationSet** that generates the same four Applications (`session-broker`, `dapr`, `keycloak`, `redis`) from a single file — a drop-in alternative to the `root-app` + `gitops/apps/` app-of-apps.

```
Argo CD
  └── ApplicationSet: session-broker-platform   (list generator)
        ├── session-broker  ──▶ gitops/session-broker/overlays/dev   (Kustomize)
        ├── dapr            ──▶ dapr/dapr Helm chart        + gitops/dapr/values.yaml
        ├── keycloak        ──▶ bitnami/keycloak Helm chart + gitops/keycloak/values.yaml
        └── redis           ──▶ bitnami/redis Helm chart    + gitops/redis/values.yaml
```

It uses a **list generator** (one element per component) rather than a Git *directory* generator, because the `gitops/` tree is heterogeneous: `dapr/`, `keycloak/`, and `redis/` are values-only folders whose charts live in remote Helm repos (multi-source `$values` ref), while `session-broker/` is a Kustomize overlay. A directory generator pointed at `gitops/*` would try — and fail — to render the values-only folders. The differing source shapes (Helm multi-source vs. Kustomize single-source) are injected via `templatePatch`, since a structured `template` can't conditionally add or remove keys.

:::warning
The ApplicationSet and the `root-app` app-of-apps both create Applications named `session-broker`, `dapr`, `keycloak`, and `redis`. **Use one or the other, not both** — running both makes two owners fight over the same Applications.
:::

## Namespaces

| Workload | Namespace |
|---|---|
| Session Broker | `session-broker` |
| Dapr control plane | `dapr-system` |
| Keycloak | `keycloak` |
| Redis | `redis` |
| Argo CD | `argocd` |

## Overlays

The `dev` overlay uses:
- `replicas: 1`
- Lower CPU/memory limits
- `image tag: latest`

The `prod` overlay uses:
- `replicas: 3`
- Higher CPU/memory limits
- Pinned image tag (e.g. `1.0.0`)

To promote to production, update the image tag in `gitops/session-broker/overlays/prod/kustomization.yaml` and push. Argo CD detects the drift and reconciles.
