# Dapr + Keycloak Integration

Session Broker uses Keycloak as the OAuth Authorization Server and IdP federation hub, and Dapr as runtime middleware plus state abstraction.

## Write path — OAuth Authorization-Code flow (demo, as of 2026-06-23)

The broker is the **confidential OAuth client**. It performs the `code → token` exchange itself; it never trusts raw tokens supplied by a caller. The Slack user ID is bound to the Keycloak/Google identity via a broker-minted, single-use `state` nonce — the caller never supplies a `slack_user_id` at the callback.

### End-to-end flow

```
EnterpriseClaw gateway
        │
        │ POST /auth/login/start { slack_user_id }  (internal-only)
        ▼
  Session Broker
        │ 1. Mint one-time nonce + PKCE code_verifier
        │ 2. Store login:{nonce} → { slack_user_id, code_verifier }  (Redis, ~5 min TTL)
        │ 3. Build Keycloak /authorize URL (state=nonce, code_challenge=S256)
        │ 4. Return { authorize_url, state }
        ▼
  Gateway posts authorize_url to user in Slack
        │
        │ User opens URL in browser
        ▼
  Keycloak  ←── Google Workspace (upstream IdP, transparent to broker)
        │  User authenticates via Google → Keycloak
        │  Keycloak issues authorization code
        │
        │ Browser redirect: GET /auth/callback?code=&state=
        ▼
  Session Broker  ← reached via AWS ALB Ingress
        │ Step 1 — Identify: consume nonce (delete-on-read) → slack_user_id + code_verifier
        │ Step 2 — Exchange: POST code + client_secret + code_verifier to Keycloak token endpoint
        │            ← access_token, refresh_token, id_token (provenance guaranteed)
        │ Step 3 — Store: AES-256-GCM encrypt token set → Redis keyed by slack:{slack_user_id}
        │            TTL = access_token exp claim; audit log emitted
        ▼
  Browser redirected to success page
```

### Keycloak confidential client requirements

| Setting | Value |
|---|---|
| Client ID | `session-broker` |
| Client secret | K8s Secret `keycloak-client-secret` |
| `redirect_uri` | `https://<alb-hostname>/auth/callback` |
| Scopes | `openid email profile offline_access` |
| PKCE method | `S256` |
| Grant type | `authorization_code` |
| Realm | `enterpriseclaw` |

The `offline_access` scope causes Keycloak to issue a refresh token alongside the access token.

### Identity binding — why the nonce design

The `state` nonce is minted by the broker (not the caller) and stored single-use in Redis. This closes two security properties simultaneously:

- **Binding**: the Slack user ID rides in the nonce store, never in the callback URL. A caller who does not know the nonce cannot inject an arbitrary `slack_user_id`.
- **Provenance**: the broker performs the `code → token` exchange, so every token in Redis is guaranteed Keycloak-issued.

:::note Google is Keycloak's upstream IdP
The broker only ever calls Keycloak. Google Workspace federation is configured inside Keycloak and is transparent to the broker's code.
:::

### ALB Ingress

The callback endpoint `GET /auth/callback` must be browser-reachable. It is exposed via a Kubernetes Ingress with AWS ALB Controller annotations, matching the EnterpriseClaw main project's ingress pattern.

:::warning Demo prerequisite
The ALB Ingress only resolves on an AWS-backed cluster running the AWS Load Balancer Controller. The current local kubeadm-in-UTM cluster has no ingress controller — the callback URL will not be reachable there.
:::

### Front-channel vs back-channel Keycloak URLs

The write path contacts Keycloak in two distinct ways. The guiding principle is **who makes the request**:

| Channel | Who originates it | Env var | Example value |
|---|---|---|---|
| **Front-channel** | User's **browser** | `KEYCLOAK_ISSUER_URL` | `https://keycloak.session-broker.example.com/realms/enterpriseclaw` |
| **Back-channel** | **Broker pod** (server-to-server) | `KEYCLOAK_TOKEN_URL` | `http://keycloak.keycloak.svc.cluster.local/realms/enterpriseclaw/protocol/openid-connect/token` |

**Front-channel** — two steps involve the user's browser:
1. `POST /auth/login/start` builds the Keycloak `/authorize` URL. The broker constructs this URL from `KEYCLOAK_ISSUER_URL` (the external host) because the URL is sent to the user in Slack and opened in a browser. It must be internet-reachable.
2. The `redirect_uri` registered with Keycloak (`https://broker.session-broker.example.com/auth/callback`) is also on the external host — Keycloak's 302 redirect sends the browser there after authentication.

**Back-channel** — step 2 of the callback (the `code → token` exchange) is a direct server-to-server POST the broker pod makes itself. It targets `KEYCLOAK_TOKEN_URL`, the Keycloak ClusterIP Service URL. This call never leaves the cluster and does not hairpin through the ALB.

```
Browser                         Broker pod (writer)              Keycloak (in-cluster)
   │                                    │                                │
   │  (1) click authorize URL           │                                │
   │ ──────────────────────────────────────────────────────────────────► │
   │                              [user authenticates]                   │
   │  (2) Keycloak 302 → /auth/callback │                                │
   │ ──────────────────────────────────►│                                │
   │                                    │  (3) POST token exchange       │
   │                                    │ ──────────────────────────────►│
   │                                    │    (back-channel, in-cluster)  │
   │                                    │ ◄──────────────────────────────│
   │                                    │    access_token, refresh_token │
```

:::info Writer security posture
The writer pod sits in a **private subnet**. It is reachable from the internet **only on `/auth/callback`** via the AWS ALB Ingress. All other communication (nonce store → Redis, token exchange → Keycloak ClusterIP) is east-west, never leaving the cluster. Internet-exposure and backend-reach are intentionally separated by this channel split.
:::

:::info KEYCLOAK_TOKEN_URL fallback
If `KEYCLOAK_TOKEN_URL` is not set, the broker derives the token endpoint by appending `/protocol/openid-connect/token` to `KEYCLOAK_ISSUER_URL`. This works in single-cluster setups but routes the exchange through the external host. Set `KEYCLOAK_TOKEN_URL` explicitly to keep the exchange in-cluster.
:::

:::warning KC_HOSTNAME must be pinned to the external value
Keycloak must have `KC_HOSTNAME` (frontend URL) set to the **external** issuer host. Because the broker POSTs to the in-cluster ClusterIP for the token exchange, Keycloak would otherwise stamp `iss` with the internal hostname — making tokens fail validation on callers that check against the external issuer. Pin `KC_HOSTNAME` so the `iss` claim is always `https://keycloak.session-broker.example.com/realms/enterpriseclaw` regardless of which host the broker contacted.
:::

## Read path — resolve cached token

The read path is **entirely Keycloak-free**. It resolves the cached token from Redis and returns the user's access token to the caller.

```
Argo Workflows step (EnterpriseClaw)
        │
        │ POST /identity/resolve
        │ X-Slack-User-Id: U07ABC123
        ▼
  Session Broker
        │ 1. Look up slack:{slack_user_id} in Redis (via Dapr state store)
        │ 2. Decrypt token set (AES-256-GCM)
        │ 3. Validate exp claim — reject if expired (401)
        │ 4. Emit audit log
        │ 5. Return access_token + identity claims
        ▼
  Caller attaches access_token as Bearer on downstream agent calls
```

The reader must never contact Keycloak. Workload trust is enforced by mTLS in EnterpriseClaw (Istio); the broker itself does not implement that boundary — its job stops at storing and returning the token.

## Authorization model — claims the brokered token carries

The broker **does not enforce** permissions; it brokers a token that *carries* them. The realm encodes the user's authorization context as standard OIDC claims, and **downstream EnterpriseClaw agents enforce** on those claims. The model is configured declaratively (see [Declarative realm configuration](../gitops/deployment)):

- **Groups** model org placement: `/engineering` (every authenticated engineer) → `/engineering/seniors` (inherits `/engineering`, adds senior capabilities).
- **Realm roles**: `agent-user` (baseline marker for authenticated users) and `senior-engineer` (composite → `infra-provisioner:db:provision:dev`).
- **Per-agent client roles**: each downstream agent is a `bearerOnly` Keycloak client holding its own roles — `issue-tracker` (`issue:create`, `issue:read`) and `infra-provisioner` (`infra:read`, `db:provision:dev`, `db:provision:prod`). They appear under `resource_access.<agent>.roles`.

One authentication yields one cached token carrying the user's roles for **many** downstream agents:

| Claim | `alice` (junior, `/engineering`) | `juandavidarguello@…` (senior, `/engineering/seniors`) |
|---|---|---|
| `groups` | `[/engineering]` | `[/engineering, /engineering/seniors]` |
| `realm_access.roles` | `[agent-user, …]` | `[agent-user, senior-engineer, …]` |
| `resource_access.issue-tracker.roles` | `[issue:create, issue:read]` | `[issue:create, issue:read]` |
| `resource_access.infra-provisioner.roles` | `[infra:read]` | `[infra:read, db:provision:dev]` |
| File an issue? | ✅ | ✅ |
| Provision a **dev** DB? | ❌ | ✅ |
| Provision a **prod** DB? | ❌ | ❌ (`db:provision:prod` granted to nobody) |

`POST /identity/resolve` returns the access token, so the caller and any downstream agent see these claims directly. The broker echoes them but makes **no** allow/deny decision of its own — keying the cache stays on the Slack user ID, and the read path remains Keycloak-free.

## Dapr integration

### State store — Redis

`gitops/session-broker/base/dapr-statestore.yaml` defines the state store used by the broker:

```yaml
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: redis
  namespace: session-broker
spec:
  type: state.redis
  version: v1
  metadata:
    - name: redisHost
      value: "redis-master.redis.svc.cluster.local:6379"
```

All state operations (nonce store, token store, token read) go through this Dapr component. The broker never calls Redis directly.

### Bearer middleware — write path only

Dapr's `middleware.http.bearer` component is used to validate inbound Keycloak tokens on protected endpoints. It is **not** applied to the read path (`/identity/resolve`) — the reader must not be forced to present a Keycloak bearer token and must not contact Keycloak.

```yaml
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: keycloak-validator
  namespace: session-broker
spec:
  type: middleware.http.bearer
  version: v1
  metadata:
    - name: jwksURL
      value: "http://keycloak.keycloak.svc.cluster.local/realms/session-broker/protocol/openid-connect/certs"
    - name: audience
      value: "session-broker"
    - name: issuer
      value: "http://keycloak.keycloak.svc.cluster.local/realms/session-broker"
```

### Dapr service invocation endpoints

```http
POST   /v1.0/invoke/session-broker/method/auth/login/start
GET    /v1.0/invoke/session-broker/method/auth/callback
POST   /v1.0/invoke/session-broker/method/identity/resolve
GET    /v1.0/invoke/session-broker/method/health
```

## Keycloak realm setup

The realm is configured **declaratively** by keycloak-config-cli — see
[Declarative realm configuration](../gitops/deployment) for the full catalog and
the `keycloak-realm-secrets` requirement.

| Setting | Value |
|---|---|
| Realm | `enterpriseclaw` |
| Client ID | `session-broker` |
| Client type | Confidential (client secret required) |
| Client scopes | `openid email profile offline_access` |
| IdP federation | Google Workspace (configured inside Keycloak) |
| `redirect_uri` | `https://<alb-hostname>/auth/callback` + `http://localhost:8000/auth/callback` (local tunnel) |
| JWKS endpoint | `http://keycloak.keycloak.svc.cluster.local/realms/enterpriseclaw/protocol/openid-connect/certs` |

## Deferred to hardening

- **Email cross-check at callback** — verify the Keycloak token's `email` claim matches the Slack user's corporate email; backstop against identity-injection on `login/start`.
- **Writer/reader split** — separate Deployments, service accounts, and Redis ACL users for the write and read paths.
- **Refresh-token usage** — broker proactively refreshing an expired access token on the read path. Design tension unresolved: the reader must not contact Keycloak, but refresh requires it.
